# Scope

## In Scope

The initial scope of the project is the operational foundation of the platform.

This means the first serious product scope is centered on building a usable, governable, extensible base for AI-native digital work.

### Core in-scope areas

- project spaces as the unit of organized work
- AI agents as structured workers with explicit roles
- teams and departments as first-class coordination constructs
- workflow modeling across core delivery stages
- orchestration of agent execution and task movement
- operator visibility into active work, blockers, and intervention points
- realtime status, monitoring, and control surfaces
- workspace-oriented product shell that can host operational views
- governance, ownership, and documentation systems for scaling the platform safely

### V1 operational scope

The first version should prove the platform's operational core.

That includes:

- creating and managing project spaces
- organizing agents into teams or departments
- representing workflow and task progression visibly
- enabling human oversight, intervention, and control
- supporting a serious orchestration model rather than isolated agent sessions
- defining a scalable platform architecture and monorepo structure

### Documentation scope

The documentation effort is also in scope as a first-class deliverable.

This includes:

- vision and platform thesis
- product identity
- organization and project modeling
- architecture and platform boundaries
- governance and worklog rules
- migration and execution strategy

The docs are not secondary support material. They are part of the product foundation.

## Out of Scope

The following items are explicitly out of scope for the initial delivery phase, even if they remain part of the long-term vision.

### Deferred strategic layers

- full marketplace implementation
- agent buying, selling, or renting flows
- production-grade economy mechanics
- monetization systems beyond early conceptual design
- public reputation markets or trust economies

### Deferred experience layers

- fully realized 3D world as a primary interface
- game-first mechanics
- heavy gamification systems
- virtual city expansion as a delivery blocker

### Deferred platform complexity

- multi-region production deployment as a launch requirement
- large-scale sharding implementation before platform fit is proven
- full enterprise-grade distributed architecture before the core operating model is validated

### Explicit non-goals for v1

- building a generic consumer AI app
- shipping a purely aesthetic virtual world with weak operational depth
- supporting every possible plugin, runtime, or workflow variation from day one
- recreating the entire legacy system before defining the new one properly
- solving the entire AI-native economy problem in the first release

### Archive policy

The legacy monorepo remains reference material only.
Restoring it as the active product is out of scope.

## Assumptions

The project currently assumes the following:

- AI-assisted work will become more organizational and less purely individual over time.
- Teams will need structured ways to operate multiple AI workers in parallel.
- Users will value control, observability, and intervention more than novelty alone.
- A platform that models teams, departments, workflows, and project spaces can create more durable value than a single-agent interface.
- The strongest initial wedge is operational usefulness, not marketplace novelty.
- Documentation-first planning will reduce architectural drift and rework.
- The long-term vision can stay broad as long as the implementation scope remains narrow and disciplined.
- A workspace-oriented shell can eventually host both operator workflows and richer organizational experiences.

These assumptions should be re-evaluated as product learning matures.

## Constraints

The project must operate within the following constraints.

### Strategic constraints

- The vision is large, but the initial build must stay disciplined.
- The platform must remain grounded in real project execution.
- The team must avoid building speculative layers before the operational core works.

### Product constraints

- The first product must solve a real organizational execution problem.
- UX direction must balance ambition with usability.
- Workspace shell thinking must not weaken operator clarity.

### Architecture constraints

- The new platform must not inherit legacy complexity blindly.
- Boundaries must be designed before implementation scales.
- The system must be extensible, but v1 should not be defined by extension surface area.

### Delivery constraints

- Docs are part of the foundation and must be treated as source of truth.
- Governance, ownership, and decision-making must be explicit.
- The platform should be buildable in layered phases rather than through uncontrolled expansion.

### Scope discipline constraint

Every proposed feature or subsystem should be evaluated with one question:

Does it strengthen the operational core of AI-native digital organizations now, or is it a later-layer ecosystem concern?

If it is a later-layer concern, it should be documented but not allowed to distort the current implementation scope.
